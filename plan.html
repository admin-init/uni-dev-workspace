1. Overview & Design Principles

uni-dev is a universal backend development framework that automates the full software development lifecycle through multi-agent orchestration. It is not tied to any single project, language, or frameworkвҖ”it is a methodology encoded into an agent system.

в‘  Specification First

OpenAPI contract written before any implementation code. The spec is the source of truth. Code must conform.

в‘Ў Test Before Code

Contract tests + unit tests written before implementation. Code is judged by whether tests pass.

в‘ў Domain Model First

Entities, aggregates, value objects, bounded contexts identified before technical design.

в‘Ј Deterministic Control

Verification gates, retry limits, iteration ordering вҖ” all handled by deterministic Python, never LLM.

в‘Ө LLM for Intelligence

Code generation, architectural decisions, specification writing вҖ” handled by LLM sub-agents.

в‘Ҙ Single Source of Truth

The Knowledge Base is the canonical representation of the codebase. Agents query it; they never trust their own memory.

3. Deterministic LangGraph Core

Critical Rule These nodes are pure Python. Zero LLM calls. They cannot be overridden, hallucinated, or skipped. ~50 lines total per node.

# Node Logic LOC
в‘  Verification Gate Reads MCP compare_api_responses() + verify_contract() + test exit code.
Returns pass or fail. LLM cannot override test results.		 ~16
в‘Ў Retry Controller Counter: attempts < 3 ? "retry" : "escalate_to_human".
LLM cannot argue for a 4th try.		 ~10
в‘ў Migration Stepper Array index: idx += 1; return "done" if idx >= len(controllers) else "next".
LLM cannot skip or reorder.		 ~11
в‘Ј Classification Router Dict lookup: routes.get(classification, default).
LLM classifies; code routes. Routing table is immutable.		 ~13

4. Knowledge Base Design

Scope: Project-specific. Each repository gets its own .uni-dev/ directory containing SQLite, ChromaDB, and graph data.

tree-sitter Parsers

Source code parsed into structured JSON via plugin-based language parsers. Shipped: Java + Node.js.

SQLite (8 tables)

Relational store for modules, classes, methods, endpoints, DB tables/columns, UI components.

ChromaDB (14 indexes)

Embedding + BM25 keyword indexes for semantic and exact search across code, specs, and docs.

NetworkX Graph

8 node types, 11+3 edge types. Dependency graph for migration planning.

5. Parser Extension System

Plugin Architecture Each language parser is a standalone plugin implementing the ParserPlugin interface. Plugins are discovered via Python entry points and loaded on demand.

ParserPlugin Interface

class ParserPlugin(ABC):
    """Base class for all language parsers."""

    @abstractmethod
    def language(self) вҶ’ str:
        """Return language identifier: 'java', 'typescript', etc."""
        ...

    @abstractmethod
    def detect(self, file_path: str) вҶ’ bool:
        """Return True if this parser should handle the file."""
        ...

    @abstractmethod
    def parse(self, file_path: str, source: str) вҶ’ ParseResult:
        """Parse source code into structured JSON."""
        ...

Shipped Parsers

Java / Spring Boot

unicore-parser-java

  • ControllerParser вҖ” extracts @RestController, @GetMapping, etc.
  • ServiceParser вҖ” extracts @Service, method signatures
  • EntityParser вҖ” extracts @Entity, @Column, JPA annotations
  • MapperParser вҖ” extracts MyBatis XML mapper queries

Node.js / Express / NestJS

unicore-parser-nodejs

  • RouteParser вҖ” extracts Express router.get(), NestJS @Get()
  • ServiceParser вҖ” extracts service classes and methods
  • ModelParser вҖ” extracts Sequelize/TypeORM model definitions
  • MiddlewareParser вҖ” extracts auth middleware chains

7. LLM Sub-Agents

domain-designer

DDD Phase

Identifies entities, aggregates, value objects, and bounded contexts from the issue description and existing codebase.

Tools: search_code, get_class_structure, get_dependency_graph

Output: Domain model YAML

spec-writer

SDD Phase

Writes or updates OpenAPI 3.0 YAML contracts based on the domain model. The spec is the source of truth.

Tools: get_api_contract, get_entity_spec, filesystem tools

Output: OpenAPI YAML

code-generator

TDD Phase

Implements code conforming to the spec. Uses test failures to guide corrections.

Tools: filesystem tools, shell (for builds/tests), get_business_logic_doc

Output: Source files

test-generator

TDD Phase

Generates contract tests and unit tests from specs. Tests are written before implementation.

Tools: get_api_contract, verify_contract, shell

Output: Test files

reviewer

Post-Verification

Reviews implementation against spec, generates documentation updates, identifies gaps.

Tools: All MCP tools, compare_api_responses, get_migration_checklist

Output: Review report + doc updates

8. Security Filter

Critical Middleware that intercepts tool call arguments and results to redact sensitive information before it reaches logs or the knowledge base. 

class LogFilter:
    """Redact PII, secrets, and credentials from agent I/O."""

    PATTERNS: list[tuple[str, str]] = [
        (r'Bearer\s+[A-Za-z0-9\-._~+/=]+',    'Bearer <REDACTED>'),
        (r'api[_-]?key[=:]\s*[^\s&]+',         'api_key=<REDACTED>'),
        (r'password[=:]\s*[^\s&]+',            'password=<REDACTED>'),
        (r'[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}', '<EMAIL>'),
        (r'postgresql://[^@]+@',                'postgresql://<CREDENTIALS>@'),
        (r'mongodb://[^@]+@',                    'mongodb://<CREDENTIALS>@'),
        (r'REDIS_URL[=:]\s*[^\s]+',             'REDIS_URL=<REDACTED>'),
    ]

    def filter_text(self, text: str) вҶ’ str:
        for pattern, replacement in self.PATTERNS:
            text = re.sub(pattern, replacement, text, flags=re.IGNORECASE)
        return text

14. Re-entrant State Machine Refactor Planned

Architectural Fix Resolves the dual-runtime problem by inverting control: the Graph becomes a pure state machine producing instructions, and the Orchestrator drives it in a loop, calling SubAgents with full tool access. Replaces CompiledSubAgent wrapping with a run_pipeline_loop() tool.

Problem

The current architecture has two separate LangGraph runtimes: (A) the deepagent orchestrator with task() and full tools, and (B) the pipeline graph wrapped as CompiledSubAgent. The graph's LLM nodes create their own ChatOpenAI instances without access to filesystem, MCP, or shell tools. The graph fabricates test_results as always-passing, making the Verification Gate a rubber stamp. Sub-agent logic is duplicated between graph.py and agents/*.py.

Solution: Re-entrant State Machine

Orchestrator Loop (run_pipeline_loop @tool)
  в”Ӯ
  в”ңв”Җв”Җ graph.invoke(state)  вҶҗ pure Python, returns instruction
  в”Ӯ     в”Ӯ
  в”Ӯ     в”ңв”Җв”Җ ClassificationRouter вҶ’ routes to first LLM node
  в”Ӯ     в”ңв”Җв”Җ _domain_designer_node   вҶ’ returns {"next_action": "CALL_DOMAIN_DESIGNER"}
  в”Ӯ     в”ңв”Җв”Җ _spec_writer_node       вҶ’ returns {"next_action": "CALL_SPEC_WRITER"}
  в”Ӯ     в”ңв”Җв”Җ _test_generator_node    вҶ’ returns {"next_action": "CALL_TEST_GENERATOR"}
  в”Ӯ     в”ңв”Җв”Җ _code_generator_node     вҶ’ returns {"next_action": "CALL_CODE_GENERATOR"}
  в”Ӯ     в”ңв”Җв”Җ VerificationGate      вҶ’ returns pass/fail/blocked
  в”Ӯ     в”Ӯ     в”ңв”Җв”Җ pass    вҶ’ Reviewer вҶ’ COMPLETE
  в”Ӯ     в”Ӯ     в”ңв”Җв”Җ fail    вҶ’ RetryController вҶ’ retry/escalate
  в”Ӯ     в”Ӯ     в””в”Җв”Җ blocked вҶ’ EscalationRouter вҶ’ rollback to SpecWriter/DomainDesigner
  в”Ӯ     в””в”Җв”Җ _reviewer_node          вҶ’ returns {"next_action": "COMPLETE"}
  в”Ӯ
  в”ңв”Җв”Җ task(subagent_name, input)  вҶҗ full tools available
  в”Ӯ     в”Ӯ
  в”Ӯ     в”ңв”Җв”Җ normal result вҶ’ merge into state вҶ’ loop
  в”Ӯ     в””в”Җв”Җ BLOCKED result вҶ’ write blockers вҶ’ loop (graph routes to EscalationRouter)
  в”Ӯ
  в””в”Җв”Җ repeat until COMPLETE or circuit breaker (max 50 iterations)

New Components

EscalationRouter (new deterministic node)

Pure Python, zero LLM. Reads state.blockers and routes to fallback target based on suggested_action:

  • REVISE_SPEC вҶ’ SpecWriter
  • REVISE_DESIGN вҶ’ DomainDesigner
  • HUMAN_REQUIRED вҶ’ END

Anti-loop protection: detects consecutive identical rollback paths (fires on 1st repetition) + global MAX_ESCALATIONS=3 circuit breaker.

run_pipeline_loop() (new orchestrator function)

Registered as a @tool on the deepagent. Has direct access to task(). Drives the graph in a while-loop:

  1. graph.invoke(state) вҶ’ instruction
  2. If COMPLETE вҶ’ return state
  3. task(subagent, input) вҶ’ parse BLOCKED or merge result
  4. Loop with MAX_LOOP_ITERATIONS=50 circuit breaker

State Schema Extension

class Blocker(TypedDict, total=False):
    reason: str
    suggested_action: str  # "REVISE_SPEC" | "REVISE_DESIGN" | "HUMAN_REQUIRED"
    blocking_details: dict

class NextAction(TypedDict):
    type: str              # "CALL_SUBAGENT" | "COMPLETE"
    subagent: NotRequired[str]
    input: NotRequired[dict]

class UniDevState(TypedDict):
    # ... existing 12 fields ...
    next_action: NotRequired[NextAction]       # instruction for orchestrator loop
    blockers: NotRequired[list[Blocker]]       # structured blocking info
    escalation_history: NotRequired[list[str]]  # rollback trajectory (anti-loop)

Node Idempotency Pattern

Each of the 5 LLM nodes becomes an instruction producer using the unified pattern: (output_exists AND NOT needs_rework) ? skip : produce instruction.

_domain_designer_nodestate.domain_model existsвҖ” (never re-triggered)

Sub-Agent BLOCKED Protocol

All 5 SubAgents standardize on a structured BLOCKED response when they encounter an unsolvable problem:

{
    "status": "BLOCKED",
    "reason": "API endpoint /v2/users/search does not exist in contract",
    "suggested_action": "REVISE_SPEC",
    "blocking_details": {...}
}

The _execute_subagent() helper in run_pipeline_loop() detects BLOCKED status and writes it to state.blockers, triggering the EscalationRouter on the next graph.invoke().

Design Decisions

  • invoke + idempotency (not stream + checkpoint) вҖ” graph is a pure stateвҶ’instruction converter. Each graph.invoke() is a complete, reproducible state transition. No checkpoint persistence needed for 9-node graph.
  • Option A: run_pipeline_loop as @tool вҖ” the loop runs inside the deepagent context where task() is natively available. No tool re-binding needed. The orchestrator LLM becomes a classifier + initiator.
  • run_pipeline_loop() in orchestrator.py (not separate module) вҖ” cohesion: the loop directly depends on the orchestrator's SubAgent registry and state merging logic. Single orchestrator scenario makes a separate module YAGNI.
  • MAX_LOOP_ITERATIONS = 50 вҖ” circuit breaker prevents infinite re-entry loops. Combined with EscalationRouter's repeated-path detection and global max, the system has defense in depth against runaway pipelines.
