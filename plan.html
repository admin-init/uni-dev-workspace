1. Overview & Design Principles

uni-dev automates the full SDLC through a layered architecture that combines deepagents' native LLM autonomy with deterministic business-semantic gates. The graph topology is a map of business semantics, not a stream of code execution.

Core Insight: The graph topology represents the DDD→SDD→TDD methodology as explicit business gates. Within each phase, the LLM has full autonomy via deepagents' native agent loop. Middleware handles implementation details (output cleaning, logging, cost tracking) without polluting the graph topology.

1. Specification First

OpenAPI contract written before implementation. The SDD gate validates spec completeness before TDD begins.

2. Test Before Code

Tests written before implementation. TDD Red phase (test-generator) precedes TDD Green phase (code-generator).

3. Domain Model First

Entities, aggregates, bounded contexts identified before technical design. DDD gate validates domain completeness.

4. DeepAgent-Native Phases

Each phase node uses deepagents' native task() delegation. LLM autonomously decides how to use tools within the phase.

5. Business-Semantic Gates

Verification gates are explicit LangGraph nodes. They are deterministic, zero-LLM, and visible in the topology.

6. Middleware for Glue

Output cleaning, token tracking, log filtering, and cost monitoring are middleware — hidden from the graph, doesn't pollute topology.

Decision Heuristic: Node vs Middleware

QuestionAnswerPlacement
Does its result affect the next step in the graph?YesExplicit Node
Does it need manual review or independent visibility in Trace?YesExplicit Node
Is it merely "cleaning/enhancement" of current node's output?YesMiddleware
Is it a global guard (budget, logging) unrelated to SOP phases?YesMiddleware

3. Pipeline SOP Graph

The pipeline is a LangGraph StateGraph that maps the DDD→SDD→TDD methodology as explicit business-semantic nodes. Each phase node invokes a deepagent sub-agent via native task() delegation.

3.3 PipelineState

class PipelineState(TypedDict):
    issue: str                          # Original issue description
    project_path: str                   # Target project directory
    classification: str                 # add_feature | update_api | remove_feature | refactor
    domain_model: NotRequired[str]      # YAML from domain-designer
    api_spec: NotRequired[str]          # OpenAPI YAML from spec-writer
    test_files: NotRequired[list[str]] # Test file paths from test-generator
    modified_files: NotRequired[list[str]] # Modified source files from code-generator
    test_results: NotRequired[dict]     # {"exit_code": int, "failures": [...], "output": str}
    attempt_count: int                  # Retry counter (max 3)
    review_report: NotRequired[str]     # Reviewer output
    blockers: NotRequired[list[dict]]  # BLOCKED reasons from sub-agents

3.4 Compilation

from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.sqlite import SqliteSaver

def compile_pipeline(checkpointer: SqliteSaver) -> CompiledStateGraph:
    graph = StateGraph(PipelineState)

    # Agent phase nodes
    graph.add_node("classify", classify_node)
    graph.add_node("domain_design", domain_design_node)
    graph.add_node("spec_write", spec_write_node)
    graph.add_node("test_generate", test_generate_node)
    graph.add_node("code_generate", code_generate_node)
    graph.add_node("run_tests", run_tests_node)
    graph.add_node("review", review_node)

    # Deterministic gate nodes
    graph.add_node("ddd_gate", ddd_verification_gate)
    graph.add_node("sdd_gate", sdd_verification_gate)
    graph.add_node("test_judge", test_result_judge)
    graph.add_node("retry_ctrl", retry_controller)
    graph.add_node("human_approval", human_approval_node)

    # Wiring
    graph.add_edge(START, "classify")
    graph.add_edge("classify", "domain_design")
    graph.add_edge("domain_design", "ddd_gate")
    graph.add_conditional_edges("ddd_gate", route_ddd_gate, {
        "pass": "spec_write",
        "fail": "human_approval",
    })
    graph.add_edge("spec_write", "sdd_gate")
    graph.add_conditional_edges("sdd_gate", route_sdd_gate, {
        "pass": "test_generate",
        "fail": "human_approval",
    })
    graph.add_edge("test_generate", "code_generate")
    graph.add_edge("code_generate", "run_tests")
    graph.add_edge("run_tests", "test_judge")
    graph.add_conditional_edges("test_judge", route_test_judge, {
        "pass": "review",
        "fail": "retry_ctrl",
    })
    graph.add_conditional_edges("retry_ctrl", route_retry, {
        "retry": "code_generate",
        "escalate": "human_approval",
    })
    graph.add_conditional_edges("human_approval", route_human, {
        "approve": "code_generate",
        "reject": END,
    })
    graph.add_edge("review", END)

    return graph.compile(checkpointer=checkpointer)

4. Sub-Agents

Each sub-agent is a deepagents SubAgent specification. Phase nodes in the graph invoke sub-agents via create_deep_agent() with the sub-agent's system prompt and tools. The LLM autonomously decides how to use tools within the phase.

domain-designer

DDD Phase · v4-pro · reasoning=high

Identifies bounded contexts, entities, aggregates, value objects. Outputs structured YAML.

Tools: MCP (search_code, get_class_structure, get_dependency_graph), filesystem

Gate: DDD Verification Gate validates output

spec-writer

SDD Phase · v4-flash

Writes OpenAPI 3.0 YAML contracts from domain model. Filesystem write.

Tools: filesystem (write_file, read_file), MCP (get_api_contract, get_entity_spec)

Gate: SDD Verification Gate validates spec

test-generator

TDD Red Phase · v4-flash

Generates contract + unit tests BEFORE implementation. Tests mirror source structure.

Tools: filesystem (write_file), shell (execute), MCP (get_api_contract)

code-generator

TDD Green Phase · v4-flash

Implements minimal code to pass all tests. Uses test failures for targeted corrections.

Tools: filesystem (write_file, edit_file), shell (execute, build, test)

Retry: Retry Controller enforces max 3 attempts

reviewer

Post-Verification · v4-pro · thinking=enabled

Reviews implementation against spec, identifies gaps, generates docs. Structured JSON report.

Tools: All MCP tools, filesystem (read_file, glob, grep)

BLOCKED Protocol All sub-agents return {"status": "BLOCKED", "reason": "...", "suggested_action": "REVISE_SPEC|REVISE_DESIGN|HUMAN_REQUIRED"} when they encounter unsolvable problems. The verification gate detects this and routes to Human Approval.

4.1 Phase Node Implementation

def domain_design_node(state: PipelineState) -> PipelineState:
    """Phase node: delegates to domain-designer sub-agent via deepagent."""
    agent = create_deep_agent(
        model=domain_model,
        system_prompt=DOMAIN_DESIGNER_PROMPT,
        tools=[...filesystem_tools, ...mcp_tools],
    )
    result = agent.invoke({
        "messages": [HumanMessage(
            f"Issue: {state['issue']}\n"
            f"Project: {state['project_path']}\n"
            f"Analyze domain entities, aggregates, bounded contexts."
        )]
    })
    state["domain_model"] = parse_domain_output(result)
    return state

5. Verification Gates

Deterministic · Zero LLM · Pure Python Verification gates are explicit LangGraph nodes in the SOP topology. They validate business-semantic requirements between phases. They are not middleware because their results directly determine the next step in the graph.

#GateValidatesLOC
1DDD GateDomain model: entities defined? relationships mapped? bounded contexts identified? No BLOCKED status?~30
2SDD GateOpenAPI spec: YAML parseable? endpoints defined? request/response schemas valid? No BLOCKED status?~30
3Test JudgeTest results: pytest exit code == 0? No failures? Captures output for retry feedback.~25
4Retry ControllerAttempt count ≤ 3. On exhaustion → escalate to Human Approval. Increments counter.~24

Gate Implementation Pattern

def ddd_verification_gate(state: PipelineState) -> PipelineState:
    """Deterministic gate: validates domain model completeness."""
    if state.get("blockers"):
        state["_gate_result"] = "fail"
        return state

    domain = state.get("domain_model", "")
    if not domain:
        state["_gate_result"] = "fail"
        return state

    # Parse YAML, check required fields
    parsed = yaml.safe_load(domain)
    if not parsed.get("entities") or not parsed.get("bounded_contexts"):
        state["_gate_result"] = "fail"
        return state

    state["_gate_result"] = "pass"
    return state

def route_ddd_gate(state: PipelineState) -> str:
    return state.get("_gate_result", "fail")

6. Human-in-the-Loop

BLOCKED / Max-Retries Only The pipeline runs autonomously by default. Human intervention is triggered only when:

  1. A verification gate fails (DDD Gate or SDD Gate)
  2. The retry controller exhausts 3 attempts (TDD loop)
  3. A sub-agent returns BLOCKED with suggested_action: "HUMAN_REQUIRED"

6.1 Interrupt Mechanism

from langgraph.types import interrupt

def human_approval_node(state: PipelineState) -> PipelineState:
    """LangGraph interrupt: pauses execution for human decision."""
    reason = _build_approval_reason(state)

    decision = interrupt({
        "type": "human_approval",
        "reason": reason,
        "phase": state.get("current_phase"),
        "options": ["approve", "reject", "retry"],
    })

    state["_human_decision"] = decision
    return state

def route_human(state: PipelineState) -> str:
    decision = state.get("_human_decision")
    if decision == "approve":
        return "code_generate"  # or appropriate retry target
    return "__end__"

6.2 Checkpoint & Resume

Persistent State via SqliteSaver

from langgraph.checkpoint.sqlite import SqliteSaver

checkpointer = SqliteSaver.from_conn_string(".uni-dev/checkpoints.db")
pipeline = compile_pipeline(checkpointer=checkpointer)

# Run with a thread ID for checkpointing
config = {"configurable": {"thread_id": "issue-42"}}
result = pipeline.invoke(initial_state, config)

# If interrupted, resume with the same thread ID
pipeline.invoke(None, config)  # Resumes from checkpoint

CLI uni-dev resume <thread_id> resumes a paused pipeline. uni-dev approve <thread_id> / reject / retry provide human decisions.

7. Middleware

Middleware handles implementation details that don't affect the graph topology. These are hidden from the SOP map and don't create branches or gates.

MiddlewarePurposeHook
MonitorMiddlewareObservability: tracks task() delegation timing, tool calls, phase durations. Writes to SQLite + structured logs.wrap_tool_call
SecurityFilterMiddlewarePII/secret redaction: 7 regex patterns for Bearer tokens, API keys, passwords, emails, DB URLs.wrap_tool_call
OutputCleanerMiddlewareLLM output format cleaning: removes markdown code block markers, fixes JSON escape errors.wrap_model_call
CostTrackingMiddlewareToken/cost budget monitoring. Tracks per-phase token usage. Warns on threshold.wrap_model_call

Why Middleware, Not Nodes? These don't affect the next step in the graph. They are "cleaning/enhancement" or "global guards" — exactly the decision heuristic from Section 1. The graph topology should remain a clean map of business semantics.

8.2 Monitor Middleware

SQLite + Logs The MonitorMiddleware intercepts all tool calls via wrap_tool_call. For task() calls, it creates TaskRun records with UUIDs, timestamps, and status tracking. Three output channels:

  1. LangGraph state: task_runs list in PipelineState
  2. SQLite persistence: MonitorStore for querying historical runs
  3. Structured logging: JSON logs for external log aggregators
