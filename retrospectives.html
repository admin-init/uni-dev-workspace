Key Takeaways for Future Milestones

  • Regex-based parsing is fragile. Nested parens, cross-method matching, and class vs. method ambiguity caused most bugs. Consider tree-sitter AST for complex parsing.
  • Test fixtures must be realistic. SQL strings with (), annotations with hasRole('ADMIN') — edge cases only surfaced through thoughtful test data.
  • Registry auto-load is a side effect. Query methods should not trigger discovery — find() is the only operation that needs full scan.
  • Token scope matters. HTTPS pushes of workflow files require workflow OAuth scope. SSH bypasses this entirely.
  • uvx for lint. uv run --with <tool> installs the whole project; uvx runs the tool in isolation.
  • Conditional deps are real. importlib_metadata is stdlib on Python ≥ 3.12, but 3.11 needs the backport explicitly.

Key Takeaways for Future Milestones

  • ChromaDB tests need real embedding engine. DefaultEmbeddingFunction downloads all-MiniLM-L6-v2 on first use (~80MB). Tests are correct but slow on first run. Consider a fixture that pre-initializes.
  • sqlite-utils auto-creates tables. The initial create_tables() call with pk= sets up the schema; subsequent opens reuse it. No separate migration step needed.
  • Graph edges are direction-aware. nx.DiGraph() edges point from source to target. HAS_METHOD goes Class→Method; ROUTES_TO goes Class→Endpoint; FK_TO goes Column→Table. Consistent naming is critical for topological_sort() to be meaningful.
  • CLI walks are I/O-bound. Path.rglob() across large projects will be slow. Consider parallel parsing in M4.
  • ChromaDB persistence path is absolute. Using tempfile.TemporaryDirectory in tests works but edge cases around ChromaDB background threads require careful teardown.

Problems & Solutions

#ProblemRoot CauseSolution
1 TypeORM @Entity({ name: 'users' }) not detected Detection regex only matched @Entity('name') or @Entity(), not object-argument syntax Broadened to @Entity\s*\( — matches any argument form
2 NestJS service methods not extracted — regex stopped at @InjectRepository() decorator Regex (\w+)\s*\( consumed decorator class name as method name Added decorator skip prefix: (?:@\w+\s*\([^)]*\)\s*)* before method name capture
3 Empty source produced phantom class/entity from filename stem No empty-source guard in parse() — filename-derived names leaked into result Added if not source.strip(): return ParseResult() at top of each parser's parse()
4 Express POST endpoint path had trailing slash (/api/users/ vs /api/users) Path join of prefix /api/users + method path / produced /api/users/ Added .rstrip("/") or "/" to _join_paths()

Key Takeaways for Future Milestones

  • JavaScript/TypeScript parsing is fundamentally different from Java. No annotations in the JVM sense — decorators, middleware chains, and module.exports require entirely different regex patterns.
  • TypeScript return types break method regex. : Promise<User> after params requires optional type-annotation skipping in the method signature pattern.
  • Empty-source guard is a universal pattern. Every parser should return empty ParseResult() on blank source. This should be codified in the ParserPlugin base class contract.
  • Sequelize and TypeORM diverge significantly. Field definitions, table naming, and relationship modeling use completely different syntax. Both need dedicated extraction paths with shared output format.

Key Takeaways

  • Generators are thin wrappers over SQLiteStore. All 6 generators query the store and format output. No ChromaDB or NetworkX needed except migration_checklist (graph) and search_code (Chroma in MCP).
  • YAML generation needs explicit type annotations. Pyright flags dict-of-dict assignments as type errors even when correct at runtime. Using dict[str, Any] silences them.
  • MCP tools are async generators. The @app.tool() decorator pattern works nicely — each tool maps directly to a store query or generator call.
  • Config catalog flattens nested YAML. server.port dotted keys make for easy MCP queries. .env parsing handles comments and inline values.
